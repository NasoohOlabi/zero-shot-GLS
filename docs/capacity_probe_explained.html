Capacity Probe, Explained Simply

In this project, the capacity probe answers one question: how many secret payload bits can ZGLS hide in one natural-looking short text while still recovering them exactly?

The Main Idea

Think of the generated comment as a small container. The capacity probe tries different secret sizes, such as 1 bit, 2 bits, 4 bits, 8 bits, 16 bits, and so on. For each size, it asks:

Can we hide this many payload bits, keep the visible text under the word limit, pass the quality check, then extract exactly the same bits back?

The capacity score is the largest payload size that succeeds. In the API response, that number is best_success.payload_bits_exact.

Payload Bits

The original paper describes hiding and extracting a secret bitstream. This implementation now follows that accounting directly for new `/hide` and `/capacity_probe` calls: the encoder embeds the raw payload bits, with no hidden length header.

total_target_bits = payload_bits_exact

So if the probe successfully hides 16 payload bits, the encoder tries to embed:

16 payload bits = 16 total target bits

For fair comparison with the paper-style idea of useful hidden information, report payload_bits_exact as capacity. Treat total_target_bits and total_used_bits as diagnostics for the same raw bitstream.

Concrete Example

Suppose we want a natural movie comment with at most 40 words. We ask the probe to try these payload sizes: 

[1, 2, 4, 8, 12, 16, 24, 32, 48, 64]

Example request

{
  "prompt": "You are writing natural short movie comments.\n\nExamples:\n- The acting was strong but the pacing felt uneven.\n- It had a few nice scenes, though the ending felt rushed.\n\nWrite one new natural short movie comment.",
  "payload_bits": "1011001110001111000011110000111100001111000011110000111100001111",
  "max_words": 40,
  "quality_max_retries": 6,
  "initial_payload_bits": 32,
  "payload_bits_candidates": [64, 32, 16, 8],
  "complete_sent": true,
  "max_new_tokens": 64,
  "threshold": 0.005,
  "temperature": 1.0,
  "temperature_alpha": 1.25,
  "max_bpw": 2
}

Imaginary trial results

Candidate payload Total target bits Generated comment Result
8 bits 8 bits "The cast is charming, though the story feels a little thin near the end." Success
16 bits 16 bits "The performances carry it well, but the final act wraps things up too quickly." Success
24 bits 24 bits "It has some memorable scenes, but the pacing drifts before the ending lands." Fail: payload mismatch
32 bits 32 bits "The movie looks great and starts strongly, but it loses focus halfway through." Fail: insufficient used bits
16 payload bits encoded
0 header bits
16 total target bits

In this example, the capacity is 16 bits, because 16 is the largest payload candidate that succeeded. The 24-bit candidate failed, so we do not claim 24 bits of capacity.

What Counts As Success?

A trial only succeeds when every condition below is true:

Check Meaning
Fully embedded The generated text carried the whole raw payload bitstream.
Word limit passed The visible generated text has word_count <= max_words.
Quality gate passed The comment is acceptable according to the project quality checks.
Extraction succeeded The decoder could read bits back from the generated text or token ids.
Payload matched exactly The recovered raw payload bits are identical to the original candidate bits.

Common Confusion

Do not add legacy header overhead to the capacity number.

The fair capacity number is the useful secret payload: 16 bits.

New outputs do not embed the old 16-bit length header.

A good report can still include diagnostic totals, but they should match the raw payload for new outputs: 

payload_bits_encoded: 16
protocol_overhead_bits: 0
total_embedded_bits: 16
decode_ok: true
